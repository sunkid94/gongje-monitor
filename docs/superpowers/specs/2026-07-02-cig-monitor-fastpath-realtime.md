# CIG 모니터 실시간 fast-path

- 날짜: 2026-07-02
- 대상: sunkid94/gongje-monitor (Oracle VM `~/cig-monitor`)
- 상태: 구현·검증 완료 (커밋 5b1ad3a, 6c8db51)

## 배경 / 문제

"기사 발행 시각과 CIG 모니터에 올라오는 시각 차이가 크다. 실시간으로 안 되나?"

측정(2026-07-02, 오늘 기사 발행→수집 지연):

| 소스 | 중앙값 | p90 | 10분내 |
|---|---|---|---|
| 구글 | 12분 | 326분(5.4h) | 41% |
| 네이버 | 4분 | 6분 | 100% |
| 직접 RSS | 4분 | 6분 | 98% |

→ 지연의 대부분은 **구글 색인 시차**(외부 요인). 우리 파이프라인이 더하는 지연은
"폴링 대기(최대 5분) + 전체 실행 시간(수 분)"이다.

## 제약 / 물리적 한계

- 뉴스사는 우리에게 push/webhook 을 주지 않는다 → **폴링**만 가능 → 초 단위 "동시"는 불가.
- GitHub Pages 는 push 후 재빌드(1~2분)가 더 붙는다(웹사이트 기준). 웹푸시 알림은 그보다 빠름.
- 줄일 수 있는 건 **우리 몫(폴링 주기 + 실행 무게)** 뿐. 구글/네이버 색인 속도는 못 줄인다.

## 목표

우리가 직접 RSS 를 구독하는 **업계·건설 전문지**(색인 시차 없음) 기사를 **발행 후 2분 내** 포착·푸시.

## 설계

기존 5분 주기 실행은 구글 전 키워드 + 원문 resolve + 전체 enrich 라 무겁고, 직접 RSS 도
그 사이클에 묶여 있었다. 이를 분리한다.

1. **경량 모드 `main.py --fast`**
   - 소스를 `crawler.fetch_trade_only(seen)` = `[source_rss]` 로 한정(구글/네이버 제외).
   - 나머지 파이프라인(enrich·seen·클러스터·dedup·관련도 게이트·웹푸시)은 전체 실행과 동일 재사용.
   - `--fast` 는 이메일 스킵(빈번 실행이므로), 새 기사 없으면 조용히 종료(로그 억제).
2. **cron 2분 주기**: `*/2 * * * * run.sh --fast --no-email`
   - `run.sh` 재사용(별도 스크립트 없음). 커밋 대상은 articles.json 만(기존과 동일).
3. **flock 공유**: fast 와 full 이 동일 `/tmp/cig-monitor.lock` 사용 → 중첩 실행 없음.
   full 이 락을 쥐면 fast 는 즉시 skip(다음 full 이 어차피 수집). fast 끼리도 직렬화.

## 비용 함정과 해결 (중요)

`source_rss` 본문폴백은 조직명이 **본문에만** 있어도 잡는다(recall↑). 일반 건설지는
"건설공제조합" 등을 본문에 흔히 언급 → 다수가 is_company 후보가 되고, 관련도 게이트가
대부분 제외한다. 그런데 **게이트 제외 기사는 seen 에 기록되지 않던** 기존 동작 탓에,
직접 RSS(URL 고정)에서는 **2분마다 같은 무관 기사를 재-enrich → API 비용 폭증** 위험.

**해결(커밋 6c8db51):** `main.py` 가 게이트 통과분(enriched)만이 아니라 **수집한 기사 전체
(new_articles)의 link 를 seen 에 기록**. Google 은 URL 이 회전하므로 재수집 시 새 URL 로
어차피 재평가되어 영향 없음. RSS(고정 URL)만 재판정이 멈춰 비용이 잡힌다.

## 검증

- 단위 테스트: `fetch_new_articles(sources=...)`, `fetch_trade_only`, 차단필터 유지 — 총 210 통과.
- 라이브: fast 수동 실행 시 직접 RSS 만 처리, 게이트가 무관기사(LX·국토연구원 등) 제외 확인.
- 비용: fast 연속 2회 — 콜드 19 API 콜 → 직후 0 콜(재-enrich 없음) 확인.
- 지연: 직접 피드 갱신 → 2분 폴 + enrich·푸시 ~15초 = **발행 후 ~2분**.

## 한계 / 범위 밖

- "2분 내"는 **직접 RSS 3곳**(기계설비신문·대한전문건설신문·한국건설신문) 한정.
  그 외 매체는 구글/네이버 색인 속도에 묶임 → 직접 피드를 늘리는 만큼 개선.
- 대한경제(dnews.co.kr)는 WAF 로 자동 접근 차단되어 직접 피드 추가 불가(네이버로 커버).
- 폴링 구조상 초 단위 실시간은 불가(설계 목표 아님).

## 직접 피드 추가 시 함정 — 배너 오염 (비즈워크 사례, 2026-07-02)

후보 273곳 프로브로 매체 확대를 시도했으나, 고가치 조합 매체(국토일보 SSL·대한경제
WAF·연합/뉴시스/파이낸셜 표준 RSS 없음)는 대부분 추가 불가였다. 단 이들은 네이버가
빠르게(중앙값 4분) 커버하므로 실질 손실은 작다.

**핵심 교훈 — 일반 매체는 직접 피드로 넣지 말 것:**
비즈워크(bizwork.co.kr)는 유효 RSS + 과거 조합기사 기여가 있어 후보였으나, 라이브
검증에서 **모든 기사(전기차·은행·식약처 등 무관 기사 포함)가 "건설공제조합"으로
오분류**됨을 발견. 원인은 **전 페이지에 상시 노출되는 배너(bannerpop)에 "건설공제조합"
문자열**이 있어 `source_rss` 본문폴백이 매 기사 본문에서 이를 매칭한 것.
→ 무관 기사가 전부 is_company 후보가 되어 매 런 enrich(관련도 게이트가 기각) → 순수
비용 낭비. 진짜 조합기사는 제목에 조합명이 있어 네이버로 이미 잡히므로 이득도 없음.
**추가하지 않고 되돌림.**

**새 직접 피드 추가 전 필수 검증:** `source_rss.fetch` 를 라이브로 돌려, 명백히
무관한 기사가 조합/카테고리 키워드로 잡히면 배너·사이트공통요소 오염이므로 제외.
전용 건설 전문지(기계설비·대한전문건설·한국건설)는 이 오염이 없어 안전.
