# 푸시 중복제거 v2 — 조직 별칭 통합 + 포함도 매칭 + 7일 창 설계

**작성일:** 2026-06-05
**상태:** 승인됨

---

## 개요

푸시 스토리 중복제거 v1(2026-06-04 배포, `push_dedup.py`)이 운영에서 **여전히 한 사건을 여러 번 알림**하는 것을 잡는다.

실측: "피치 A+ 유지" 단일 사건이 v1 배포 후에도 **6번** 푸시됨.

| 발송 | lead(원본) | 제목 일부 |
|------|-----------|----------|
| 6/4 20:12 | `k finco` | K-FINCO, 피치 신용등급 A+ 유지…6.5조 자본력 |
| 6/4 20:32 | `전문건설공제조합` | 전문건설공제조합, 피치 신용등급 A+ 유지 |
| 6/4 22:37 | `전문건설공제조합` | …피치 국제신용등급 A+…자본력 탄탄 |
| 6/5 00:07 | `전문건설공제조합` | …글로벌 신용평가사 피치 국제신용등급 A+ |
| 6/5 09:02 | `전문조합` | 전문조합, 보험지급능력 A+ 유지 |
| 6/5 13:57 | `k finco` | K-FINCO, 글로벌 신용평가사 피치 국제신용등급 A+ |

**누수 원인 3가지가 겹침:**
1. **같은 조직, 다른 이름** — `K-FINCO`=`전문조합`=`전문건설공제조합`(다 한 곳)인데 v1의 lead 앵커가 다른 조직으로 보고 안 묶음.
2. **수식어 붙은 헤드라인** — 같은 `전문건설공제조합`이라도 "자본력 탄탄", "글로벌 신용평가사" 등이 붙어 Jaccard가 0.5로 떨어져 0.6 문턱을 못 넘음(20:32/22:37/00:07 따로 나감).
3. **24h 창 만료** — 사건이 어제→오늘 이어지며 `k finco`가 20:12, 24h 뒤 13:57 또 발송(← 사용자가 방금 받은 알람).

> **참고(발행 지연):** 13:57 기사는 발행 08:00, 수집 13:57(구글뉴스 색인 ~6h 지연). 시스템은 *수집 시점*에 푸시하므로 "발행 몇 시간 전 기사가 방금 알림"이 됨. 그러나 근본 해결은 발행일 필터가 아니라 **중복제거 정확도**다 — 같은 스토리를 7일 창 안에서 제대로 묶으면 재유입돼도 억제된다.

## 목표 / 비목표

**목표**
- 같은 사건은 **이름·표현이 달라도** 묶어 7일 내 1번만 푸시.
- 같은 조직의 별칭(K-FINCO/전문조합/전문건설공제조합)을 하나로 취급.
- 같은 표현의 재보도/재유입(특히 사용자가 받은 13:57 K-FINCO 알람)을 억제. 0.7 임계값에서 13:57판은 00:07 전문건설판과 canon·overlap(0.86) 일치 → 억제됨.
- 단, 같은 등급이라도 **표현이 전혀 다른 헤드라인**("전문조합, 보험지급능력 A+ 유지" — 공유 토큰이 'a+/유지'뿐)은 overlap 0.7에서 별개로 남을 수 있다(보수적 = 발송). 즉 이 사건은 6회→1~2회로 감소. 임계값은 튜닝 가능.

**비목표**
- 발행일 기반 필터(별도 메커니즘, 이번 범위 밖 — 중복제거 개선으로 충분).
- 이메일·웹·저장 로직 변경(불변).
- v1의 기본 구조(`pushed.json`, 원자적 쓰기, 발송가능 확인 순서) 변경 — 유지.

**불변 안전성:** 별칭 목록에 명시된 조직끼리만 묶는다. `기계설비건설공제조합`(우리 조합)은 `전문건설공제조합`과 다른 그룹이므로 **우리 조합 뉴스가 형제 조합에 묻히지 않는다**(v1 lead 앵커가 지키던 안전성 유지).

## 컴포넌트 (push_dedup.py)

### `ORG_ALIASES` (편집 쉬운 상수)
```python
# 같은 조직의 다른 표기 — 여기에 추가하면 묶임 (모두 소문자/정규화 형태로 비교)
ORG_ALIASES = {
    "전문건설공제조합": ["전문조합", "k finco", "kfinco", "k-finco"],
    "기계설비건설공제조합": ["cig", "기계설비공제조합"],   # 우리 조합
}
```

### `canonical_org(title: str) -> str`
`story_lead(title)`로 선두 조직명을 뽑은 뒤, `ORG_ALIASES`의 (대표 또는 별칭) 문자열이 lead에 **포함되면 대표 이름**을 반환. 어디에도 안 걸리면 lead 원본 반환. 비교는 정규화(소문자·공백정리)된 문자열의 containment. 별칭은 더 구체적인(긴) 것부터 검사해 오매칭 방지.

### `overlap(a: set, b: set) -> float`
포함도 계수 = `|a∩b| / min(|a|, |b|)`. 둘 중 하나라도 비면 0. (짧은 제목이 긴 제목에 거의 포함되는 "수식어 변형"을 Jaccard보다 잘 잡음.)

### 매칭 임계값 / 창
`OVERLAP_THRESHOLD = 0.7`, `WINDOW_HOURS = 168` (7일). 모듈 상수로 노출(튜닝 가능).

### `filter_unpushed(company_articles, now)` 변경
각 기사에 대해:
- `canon = canonical_org(title)`, `key = story_key(title)`
- 억제 조건: 어떤 accepted 엔트리 `e`에 대해 **`e["canon"] == canon` 그리고 `overlap(key, e["tokens"]) >= OVERLAP_THRESHOLD`**
- 통과 시 `to_push`에 넣고, 키가 있으면 `{"tokens": key, "canon": canon, "pushed_at": now_iso, "title": title}` 기록
- (`load_pushed`/`save_pushed`는 `lead` 필드를 `canon`으로 대체. 그 외 24h→168h 창 적용 외 동일.)

빈 키 기사는 v1처럼 안전쪽 발송(기록 안 함).

## 데이터 흐름 / 호환

- `pushed.json` 엔트리: `{"tokens": [...], "canon": "전문건설공제조합", "pushed_at": "...", "title": "..."}`. `lead`→`canon` 키 변경. pushed.json은 gitignore·임시 파일이라 마이그레이션 불필요(없는 키는 기본값으로 흡수, 7일 내 자연 갱신).
- 발송가능(구독자·VAPID) 확인 → 필터 순서, 원자적 쓰기, 이메일/웹 불변 — v1 유지.

## 엣지 케이스

- **우리 조합 보호:** `canonical_org("기계설비건설공제조합, …")` → "기계설비건설공제조합", `canonical_org("전문건설공제조합, …")` → "전문건설공제조합". canon이 다르면 overlap 무관하게 억제 안 함.
- **같은 조직 다른 사건:** "전문건설공제조합 ESG 최우수" vs "전문건설공제조합 피치 A+ 유지" → canon 같지만 overlap 낮음(공통 토큰 조직명 1개뿐) → 억제 안 함.
- **별칭 오매칭 방지:** containment 검사는 별칭을 긴 것부터. 목록에 없는 약칭은 안 묶임(보수적 = 발송).
- **빈 키/손상 pushed.json:** v1 동작 유지(안전쪽 발송).
- 모든 시각 tz-aware.

## 테스트 (TDD)

`tests/test_push_dedup.py`에 추가:

1. **canonical_org** — `전문조합`/`K-FINCO`/`k-finco`(+매체접미사) → `전문건설공제조합`; `CIG`·`기계설비건설공제조합` → `기계설비건설공제조합`; 목록에 없는 조직 → lead 원본.
2. **overlap** — 포함 관계 짧은⊂긴 = 높은 값; 정확값(교집합/min) 검증; 빈 집합 0.
3. **피치 변형 감소 + 재탕 억제** — 실제 5개 제목(K-FINCO/전문건설, 수식어 변형 포함)을 한 배치에 넣으면 **건수가 줄어듦**(0.7에서 표현이 크게 다른 각도는 별개로 남을 수 있어 정확히 1로 단언하지 않음). 그리고 별도 호출로 13:57 K-FINCO 재보도가 00:07 전문건설판에 의해 **억제**됨(사용자 실측 케이스).
4. **우리 조합 분리** — `기계설비건설공제조합`(또는 `CIG`) vs `전문건설공제조합` 같은 등급 이벤트 → 둘 다 push.
5. **같은 조직 다른 사건 분리** — `전문건설공제조합` 피치 vs ESG → 둘 다 push.
6. **7일 창** — 같은 스토리 6일 뒤 → 억제, 8일 뒤 → 재push.
7. v1 기존 테스트는 임계값/창/필드명 변경에 맞춰 갱신(동작 의미는 보존: 같은 조직 동일 사건 억제, 교차조직 분리).

실패 테스트 먼저 작성 → 구현 → 통과. 임계값(0.7)·overlap 방식은 위 3·5 테스트가 동시에 통과하도록 보정.

## 예상 효과

피치 사건 6회 → **약 3회** (overlap 0.7: 같은 표현 재탕·재보도는 억제되나, 표현이 크게 다른 각도—"신용등급"vs"국제신용등급", "보험지급능력 A+"—는 별개로 남음). **사용자가 받은 13:57 K-FINCO 알람은 억제**(00:07 전문건설판과 overlap 0.86, 7일 창 안).

**등급변화 보존(중요):** overlap 0.7만으로는 부족하다 — "A+ 상향" vs "A+ 유지"는 조직·피치·신용등급·A+를 공유해 overlap 0.8로 묶여, 진짜 등급변화가 직전 "유지" 알림에 묻힐 수 있다. 그래서 **방향 가드**(`RATING_DIRECTION_TOKENS = {상향, 하향, 유지, 강등, 상승, 하락}`)를 둔다: 두 스토리가 방향 토큰을 둘 다 갖고 서로 다르면 같은 조직·유사 제목이라도 **다른 사건으로 보고 억제하지 않는다**. (둘 다 "유지"면 그대로 병합 → 13:57 케이스 보존.)
